## 阶段计划

### 阶段 1：文档与骨架

- 创建扩展内需求文档和开发计划。
- 创建 package、manifest、Vite、tsup、tsconfig、eslint、HTML 入口。
- 创建 canvas JSON 导入/导出兼容工具和测试。

### 阶段 2：后端 V1

- 创建 TypeORM 实体：项目、画布、资产库、提示词库、Provider、Workflow、Task。
- 创建服务层：默认库初始化、画布保存、软删除、导入导出、基础 CRUD。
- 创建 Web/Console 控制器，路径遵循 BuildingAI 扩展规范。

### 阶段 3：前端 V1

- 创建 React 入口、路由、i18n 和基础样式。
- 创建首页、画布列表、经典画布、智能画布、资产库、Provider 设置、ComfyUI 设置页面。
- 创建画布状态管理、节点卡片、工具栏、侧边面板、导入导出控件。

### 阶段 4：验证与补齐

- 运行兼容测试、类型检查、前后端构建。
- 修复构建问题。
- 更新本开发计划状态、根目录 `docs/todo/26.1.1-开发计划.md` 和最终变更清单。

## 每次开发任务完成后的固定动作

1. 更新本文件“任务状态表”的对应编号；若是新增任务则新增编号，不复用旧编号。
2. 状态必须与验证结果匹配：未运行构建或测试时只能写“已完成”或“进行中”，不能写“已验证”。
3. “相关文件”只列本任务实际新增或修改的主要文件；跨端任务可列目录，但需能定位到模块。
4. “验证结果”必须写明已执行命令、只读检查方式或“未涉及代码，无需构建”。
5. “遗留事项”必须写清楚剩余风险、待验证项或明确写“无”。
6. 如果任务属于阶段性功能或文档交付，同步更新根目录 `docs/todo/26.1.1-开发计划.md`，并说明本文件是 Infinite Canvas 的状态源。
7. 运行与任务相关的最小验证命令。
8. 交付前执行 `git status --short`，确认变更范围。
