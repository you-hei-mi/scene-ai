# 知识库治理

本文说明知识库从创建、导入、向量化、检索到运营治理的完整流程。

## 核心模块

知识库后端模块位于：

```text
packages/api/src/modules/ai/datasets
```

主要能力：

- 知识库管理。
- 成员管理。
- 文档上传和 URL 导入。
- 文档解析。
- 文本分段。
- 向量化。
- 检索。
- 知识库问答。
- 发布到知识广场。

## 文档生命周期

1. 创建知识库。
2. 上传文档或导入 URL。
3. 解析文档内容。
4. 按配置进行文本分段。
5. 投递 `vectorization` 队列。
6. 调用 embedding 模型生成向量。
7. 写入分段向量和处理状态。
8. 在问答或 Agent 中触发检索。
9. 根据引用片段生成回答。

## 向量化链路

关键服务：

- `VectorizationTriggerService`: 投递向量化任务。
- `VectorizationProcessor`: 消费 `vectorization` 队列。
- `VectorizationRunnerService`: 执行解析、分段、embedding 和状态写回。
- `SegmentationService`: 文本清洗和分段。

队列依赖 Redis。若文档长时间未完成，应检查 Redis、队列消费进程、模型密钥和 embedding 模型状态。

## 检索模式

知识库检索服务支持：

- `vector`: 向量相似度检索。
- `fullText`: PostgreSQL 全文检索。
- `hybrid`: 向量和全文混合检索。
- `keyword`: 关键词匹配检索。

常见参数：

- `topK`: 返回片段数量。
- `scoreThreshold`: 分数阈值。
- `scoreThresholdEnabled`: 是否启用阈值过滤。
- `semanticWeight`: 语义权重。
- `keywordWeight`: 关键词权重。
- `rerankConfig`: 是否使用 rerank 模型重排。

## 分段治理

分段质量直接影响检索效果。

建议：

- 单段不要过短，否则上下文不足。
- 单段不要过长，否则召回不精确且 token 成本高。
- 对 FAQ、制度、合同、产品文档使用相对稳定的标题或段落分隔。
- 对中文文档检查全角标点、换行和表格解析结果。
- 导入后抽查前 5 个文档的分段内容和长度。

## 模型治理

知识库依赖模型配置：

- embedding 模型必须启用。
- embedding 模型对应 Provider 必须启用。
- Provider 绑定的 Secret 必须可用。
- rerank 模型可选，启用后需要确认模型支持重排能力。

模型或密钥变更后，建议抽样重跑文档向量化并测试召回效果。

## 权限与发布

知识库权限由成员、创建者和后台审核共同控制。

治理建议：

- 私有知识库只授权必要成员。
- 发布到知识广场前检查敏感内容。
- 删除成员前确认业务影响。
- Agent 绑定知识库前确认知识库状态和成员权限。

## 运营巡检清单

- 是否存在长时间处理中的文档。
- 是否存在向量化失败文档。
- `vectorization` 队列是否积压。
- embedding Provider 是否可用。
- Redis 是否健康。
- PostgreSQL 是否有慢查询或磁盘压力。
- 检索 topK 和阈值是否适合当前场景。
- 回答是否包含可靠引用。

## 故障处理

- 文档未解析: 检查上传文件类型、文件大小、解析服务日志。
- 文档未向量化: 检查 Redis、队列、embedding 模型和 Secret。
- 检索无结果: 降低阈值、增大 topK、检查分段是否为空。
- 回答不准确: 调整分段、增加高质量文档、开启 rerank 或调整权重。
- 中文全文检索异常: 检查 PostgreSQL 扩展和回退检索路径。
