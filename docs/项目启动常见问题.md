## 1. 环境要求

### 1.1 操作系统

| 项目 | 要求 |
|------|------|
| 操作系统 | Ubuntu 24.04+ / Debian 12+ / macOS 14+ |
| 内核 | Linux 5.15+ |

### 1.2 核心依赖

| 软件 | 最低版本 | 推荐版本 | 说明 |
|------|---------|---------|------|
| Node.js | 22.20.x | 22.20.x | 项目 engine 要求，不支持 v23+ |
| pnpm | 9.x | 10.x | 包管理器 |
| PostgreSQL | 16.x | 16.x | 数据库 |
| Redis | 7.x | 8.x | 缓存 |
| Git | 2.40+ | — | 版本控制 |

### 1.3 PostgreSQL 扩展

| 扩展 | 是否必需 | 说明 |
|------|---------|------|
| `vector` (pgvector) | **必需** | 向量检索，用于 AI 功能 |
| `zhparser` | 可选 | 中文全文搜索分词，开发环境可跳过 |

### 1.4 环境变量

项目使用 `.env` 文件管理环境变量。**重要：** `.env` 文件必须放在项目根目录（`/workspace/.env`），而非 `packages/api/` 下。

环境变量由 `packages/@buildingai/config/src/utils/env.ts` 加载：

```typescript
// 从项目根目录加载 .env 文件
dotenv.config({
    path: resolve(__dirname, "../../../../../.env"),
    quiet: true,
});
```

`__dirname` 在编译后为 `packages/@buildingai/config/dist/utils/`，向上 5 级到达项目根目录。

关键环境变量：

```bash
# 数据库
DB_TYPE=postgres          # 数据库类型，当前仅支持 postgres
DB_HOST=localhost         # 数据库主机
DB_PORT=5432              # 数据库端口
DB_USERNAME=postgres      # 数据库用户名
DB_PASSWORD=postgres      # 数据库密码
DB_DATABASE=buildingai    # 数据库名
DB_DEV_SYNCHRONIZE=true   # 开发环境自动同步表结构

# Redis
REDIS_HOST=localhost      # Redis 主机
REDIS_PORT=6379           # Redis 端口

# 服务端口
SERVER_PORT=4090          # API 后端端口

# JWT
JWT_SECRET=buildingai     # JWT 签名密钥
```

### 1.5 文件监视器限制（Linux）

在 Linux 环境中，Node.js 文件监视器受 `fs.inotify.max_user_watches` 限制。本项目（Nuxt 4 + NestJS）需要同时监视大量文件，建议设置：

```bash
echo 65536 | sudo tee /proc/sys/fs/inotify/max_user_watches
```

如果无法修改系统限制（如容器/沙箱环境），需要 patch Nuxt CLI 内部代码（见 [问题 6](#问题-6-web-前端-enospc-文件监视器限制)）。

## 2. 详细启动步骤

### 2.1 前置准备

```bash
# 1. 进入项目目录
cd /workspace

# 2. 安装依赖
pnpm install

# 3. 配置环境变量（从模板创建）
cp .env.example .env
# 确认 .env 在项目根目录，不是 packages/api/
```

### 2.2 启动数据库和缓存

```bash
# 方式一：Docker（推荐）
docker compose up -d postgres redis

# 方式二：本地安装
sudo apt-get update
sudo apt-get install -y postgresql postgresql-client redis-server

# 启动 PostgreSQL
sudo pg_ctlcluster 16 main start

# 创建数据库
sudo -u postgres psql -c "ALTER USER postgres PASSWORD 'postgres';"
sudo -u postgres psql -c "CREATE DATABASE buildingai;"

# 安装 pgvector 扩展
sudo apt-get install -y postgresql-16-pgvector

# 重启 PostgreSQL 加载扩展
sudo pg_ctlcluster 16 main restart
sudo -u postgres psql -c "CREATE EXTENSION IF NOT EXISTS vector;"

# 启动 Redis
redis-server --daemonize yes
redis-cli ping  # 预期返回 PONG
```

### 2.3 构建内部依赖包

**重要：** API 依赖多个 `@buildingai/*` 内部包，必须先构建。

```bash
# 构建所有核心内部包
pnpm run build \
  --filter "@buildingai/core" \
  --filter "@buildingai/types" \
  --filter "@buildingai/utils" \
  --filter "@buildingai/constants" \
  --filter "@buildingai/errors" \
  --filter "@buildingai/config" \
  --filter "@buildingai/db" \
  --filter "@buildingai/base" \
  --filter "@buildingai/decorators" \
  --filter "@buildingai/cache" \
  --filter "@buildingai/pipe" \
  --filter "@buildingai/dict" \
  --filter "@buildingai/alipay-sdk" \
  --filter "@buildingai/wechat-sdk" \
  --filter "@buildingai/llm-file-parser" \
  --filter "@buildingai/ai-toolkit" \
  --filter "@buildingai/ai-sdk" \
  --filter "@buildingai/upgrade" \
  --filter "@buildingai/di" \
  --filter "@buildingai/dto" \
  --filter "@buildingai/logger"

# 注意：不要构建 @buildingai/web-vue，它使用 tsup 打包 Vue SFC 组件会失败
# Nuxt 开发模式会直接处理源码，无需预构建
```

### 2.4 启动 API 后端

```bash
# 方式一：使用根目录脚本
pnpm run dev:api

# 方式二：直接启动
cd packages/api && CHOKIDAR_USEPOLLING=true pnpm run dev
```

`CHOKIDAR_USEPOLLING=true` 用于解决文件监视器数量限制问题。

预期输出：
```
✔  PgSQL Status:          Connected
✔  Redis Status:          Connected
✔  Database Init:         Successful
➜  Local:                 http://localhost:4090
➜  Startup Time:          ~3000 ms
```

### 2.5 启动 Web 前端

```bash
# 方式一：使用根目录脚本
pnpm run dev:web

# 方式二：直接启动
cd apps/web-nuxt && CHOKIDAR_USEPOLLING=true pnpm run dev
```

预期输出：
```
●  Nuxt 4.4.8 (with Nitro 2.13.4, Vite 8.0.0 and Vue 3.5.39)
  ➜ Local:    http://localhost:3001/
```

### 2.6 验证服务

```bash
# 验证 API 后端
curl http://localhost:4090/consoleapi/health
# 预期返回：{"code":20000,"message":"ok","data":{"status":"ok",...}}

# 验证 Web 前端
curl -o /dev/null -w "%{http_code}" http://localhost:3001/
# 预期返回：302（重定向到登录页）
```

## 3. 问题记录与解决方案

### 问题 1：数据库驱动未指定

**错误信息：**

```
MissingDriverError: Wrong driver: "undefined" given.
Supported drivers are: "postgres", "mysql", "sqlite", ...
```

**原因分析：**

`DB_TYPE` 环境变量未设置。`packages/@buildingai/config/src/configs/db.config.ts` 中 `type: process.env.DB_TYPE as "postgres"` 读取到 `undefined`。

根本原因是 `.env` 文件放在了错误的位置。`env.ts` 从项目根目录加载 `.env`，而非 `packages/api/` 目录。

**解决方案：**

```bash
# 确保 .env 在项目根目录
cp .env.example /workspace/.env
# 不是 cp .env.example /workspace/packages/api/.env
```

**验证：** TypeORM 应能正确连接数据库，日志显示 `PgSQL Status: Connected`。

### 问题 2：内部包未构建

**错误信息：**

```
Error: Cannot find module '/workspace/packages/api/node_modules/@buildingai/core/dist/modules/index.js'
Error: Cannot find module '@buildingai/llm-file-parser'
Error: Cannot find module '@buildingai/ai-toolkit'

**原因分析：**

`@buildingai/web-vue` 构建会失败（Vue SFC 需要特殊 loader），但此包供 Nuxt 直接引用源码，无需构建。

**解决方案：**

pnpm run build \
  --filter "@buildingai/core" \
  --filter "@buildingai/types" \
  --filter "@buildingai/utils" \
  --filter "@buildingai/constants" \
  --filter "@buildingai/errors" \
  --filter "@buildingai/config" \
  --filter "@buildingai/db" \
  --filter "@buildingai/base" \
  --filter "@buildingai/decorators" \
  --filter "@buildingai/cache" \
  --filter "@buildingai/pipe" \
  --filter "@buildingai/dict" \
  --filter "@buildingai/alipay-sdk" \
  --filter "@buildingai/wechat-sdk" \
  --filter "@buildingai/llm-file-parser" \
  --filter "@buildingai/ai-toolkit" \
  --filter "@buildingai/ai-sdk" \
  --filter "@buildingai/upgrade" \
  --filter "@buildingai/di" \
  --filter "@buildingai/dto" \
  --filter "@buildingai/logger"
```

**验证：** 检查关键包的 `dist/` 目录是否存在：

```bash
ls -la packages/@buildingai/core/dist/modules/index.js
ls -la packages/@buildingai/llm-file-parser/dist/index.js
ls -la packages/@buildingai/ai-toolkit/dist/index.js
```

### 问题 3：pgvector 扩展未安装

**错误信息：**

```
QueryFailedError: Could not open extension control file
"/usr/share/postgresql/16/extension/vector.control": No such file or directory.
```

**原因分析：**

`ExtensionSeeder` 在执行 `CREATE EXTENSION IF NOT EXISTS vector;` 时失败。pgvector 是 AI 向量检索功能的必需扩展，默认未随 PostgreSQL 安装。

**解决方案：**

```bash
# 安装 pgvector
sudo apt-get install -y postgresql-16-pgvector

# 重启 PostgreSQL 以加载扩展
sudo pg_ctlcluster 16 main restart

# 手动创建扩展
sudo -u postgres psql -c "CREATE EXTENSION IF NOT EXISTS vector;"
```

**验证：** 日志中应显示 `✔ ExtensionSeeder: pgvector extension initialized`

### 问题 4：zhparser 扩展不可用

**错误信息：**

```
QueryFailedError: extension "zhparser" is not available
Could not open extension control file
"/usr/share/postgresql/16/extension/zhparser.control": No such file or directory.
```

**原因分析：**

zhparser 是中文全文搜索分词扩展，需要从源码编译安装，不在 Ubuntu 官方 apt 源中。`ExtensionSeeder` 的原代码在 zhparser 初始化失败时会 `throw error`，导致整个启动流程中断。

**解决方案：**

修改 `packages/@buildingai/db/src/seeds/seeders/extension.seeder.ts`，使 zhparser 初始化失败时不抛出异常：

```typescript
// 修改前（第 112-116 行）
} catch (error) {
    this.logError(`zhparser initialization failed: ${error.message}`);
    throw error;  // ← 这里抛出异常导致启动失败
}

// 修改后
} catch (error) {
    this.logError(`zhparser initialization failed: ${error.message}. Skipping zhparser setup (non-critical for development).`);
    // Don't throw - zhparser is not critical for development
}
```

修改后必须重新构建 db 包：

```bash
pnpm run build --filter "@buildingai/db"
```

**验证：** 日志应显示 `zhparser initialization failed: ... Skipping zhparser setup (non-critical for development).` 然后继续执行后续 seeder。

### 问题 5：extensions.json 缺失

**错误信息：**

```
ExtensionSeeder: Application extension initialization failed:
Unable to find extensions.json in project root extensions folder

DatabaseInitService: ❌ Database initialization failed:
Unable to find extensions.json in project root extensions folder
```

**原因分析：**

`ExtensionSeeder.loadExtensionsConfig()` 方法在找不到 `extensions/extensions.json` 时抛出异常，导致整个数据库初始化流程失败。

**解决方案：**

修改 `packages/@buildingai/db/src/seeds/seeders/extension.seeder.ts`，使配置文件缺失时返回空配置而非抛出异常：

```typescript
// 修改前（第 124-135 行）
private async loadExtensionsConfig(): Promise<...> {
    const possiblePath = path.join(process.cwd(), "../../extensions/extensions.json");
    if (fse.pathExistsSync(possiblePath)) {
        return await fse.readJson(possiblePath);
    }
    throw new Error("Unable to find extensions.json in project root extensions folder");
}

// 修改后
private async loadExtensionsConfig(): Promise<...> {
    const possiblePath = path.join(process.cwd(), "../../extensions/extensions.json");
    if (fse.pathExistsSync(possiblePath)) {
        return await fse.readJson(possiblePath);
    }
    this.logInfo("extensions.json not found, skipping application extension initialization");
    return { applications: {}, functionals: {} };
}
```

修改后重新构建：

```bash
pnpm run build --filter "@buildingai/db"
```

### 问题 6：Web 前端 ENOSPC 文件监视器限制

**错误信息：**

```
ENOSPC: System limit for number of file watchers reached,
watch '/workspace/apps/web-nuxt/.nuxt/dist'

at FSWatcher.<computed> (node:internal/fs/watchers:316:19)
at #initializeNuxt (...@nuxt/cli/dist/dev-CQg8KtEQ.mjs:543:23)
```

**原因分析：**

Nuxt 4 的 CLI 内部使用原生 `fs.watch`（而非 `chokidar`）来监视文件变化。Linux 环境中默认的 `inotify` 监视器数量有限（通常为 8192 或 15173），项目文件数量远超此限制。

`CHOKIDAR_USEPOLLING=true` 环境变量仅对使用 chokidar 的工具（如 nodemon）生效，Nuxt CLI 内部直接调用 `import { watch } from "node:fs"`，不受此环境变量影响。

**解决方案：**

**方案 A（推荐）：** 增加系统限制

```bash
echo 65536 | sudo tee /proc/sys/fs/inotify/max_user_watches
```

**方案 B（容器/沙箱环境）：** Patch Nuxt CLI

当系统文件为只读时，需要 patch Nuxt CLI 内部代码，将所有 `fs.watch()` 调用包装在 try-catch 中：

文件路径：`node_modules/.pnpm/@nuxt+cli@3.36.1_.../node_modules/@nuxt/cli/dist/dev-CQg8KtEQ.mjs`

修改位置 1（第 543 行，dist watcher）：

```javascript
// 修改前
this.#distWatcher = watch(distDir);
this.#distWatcher.on("change", (_event, file) => { ... });

// 修改后
try {
  this.#distWatcher = watch(distDir);
  this.#distWatcher.on("change", (_event, file) => { ... });
} catch(_) { /* ENOSPC - file watcher limit reached, skipping watcher */ }
```

修改位置 2（第 641 行，config dir watcher）：

```javascript
// 修改前
const watcher = watch(dir);

// 修改后
let watcher;
try { watcher = watch(dir); } catch(_) { return () => {}; }
```

修改位置 3（第 659 行，.config dir watcher）：

```javascript
// 修改前
const configDirWatcher = watch(configDir);

// 修改后
let configDirWatcher;
try { configDirWatcher = watch(configDir); } catch(_) { return () => {}; }
```

此外，在各终端中启动时均需设置 `CHOKIDAR_USEPOLLING=true`：

```bash
CHOKIDAR_USEPOLLING=true pnpm run dev:api
CHOKIDAR_USEPOLLING=true pnpm run dev:web
```

## 4. 预防措施与建议

### 4.1 自动化启动脚本

建议创建 `scripts/dev-start.sh` 一键启动脚本：

```bash
#!/bin/bash
set -e

echo "=== BuildingAI 开发环境启动 ==="

# 1. 检查依赖
echo "[1/5] 检查系统依赖..."
command -v node >/dev/null 2>&1 || { echo "需要安装 Node.js"; exit 1; }
command -v pnpm >/dev/null 2>&1 || { echo "需要安装 pnpm"; exit 1; }
command -v psql >/dev/null 2>&1 || { echo "需要安装 PostgreSQL"; exit 1; }
command -v redis-cli >/dev/null 2>&1 || { echo "需要安装 Redis"; exit 1; }

# 2. 环境变量
echo "[2/5] 配置环境变量..."
if [ ! -f .env ]; then
    cp .env.example .env
    echo "  .env 已从 .env.example 创建"
fi

# 3. 确保数据库和缓存运行
echo "[3/5] 检查数据库和缓存..."
pg_isready -q || { echo "  PostgreSQL 未运行，请先启动"; exit 1; }
redis-cli ping >/dev/null 2>&1 || { echo "  Redis 未运行，请先启动"; exit 1; }

# 4. 构建内部包
echo "[4/5] 构建内部依赖包..."
pnpm run build \
  --filter "@buildingai/core" \
  --filter "@buildingai/types" \
  --filter "@buildingai/utils" \
  --filter "@buildingai/constants" \
  --filter "@buildingai/errors" \
  --filter "@buildingai/config" \
  --filter "@buildingai/db" \
  --filter "@buildingai/base" \
  --filter "@buildingai/decorators" \
  --filter "@buildingai/cache" \
  --filter "@buildingai/pipe" \
  --filter "@buildingai/dict" \
  --filter "@buildingai/alipay-sdk" \
  --filter "@buildingai/wechat-sdk" \
  --filter "@buildingai/llm-file-parser" \
  --filter "@buildingai/ai-toolkit" \
  --filter "@buildingai/ai-sdk" \
  --filter "@buildingai/upgrade" \
  --filter "@buildingai/di" \
  --filter "@buildingai/dto" \
  --filter "@buildingai/logger"

# 5. 设置文件监视器限制
echo "[5/5] 设置文件监视器限制..."
if [ -w /proc/sys/fs/inotify/max_user_watches ]; then
    echo 65536 | sudo tee /proc/sys/fs/inotify/max_user_watches > /dev/null
    echo "  max_user_watches 已设置为 65536"
else
    export CHOKIDAR_USEPOLLING=true
    echo "  无法修改系统限制，已启用 CHOKIDAR_USEPOLLING"
fi

echo ""
echo "=== 环境就绪 ==="
echo "启动 API 后端: pnpm run dev:api"
echo "启动 Web 前端: pnpm run dev:web"
echo "API 健康检查: curl http://localhost:4090/consoleapi/health"
```

### 4.2 配置检查清单

在启动项目前，确认以下项目：

- [ ] `.env` 文件存在于项目根目录（`/workspace/.env`）
- [ ] `DB_TYPE` 设置为 `postgres`
- [ ] `DB_HOST`、`DB_PORT`、`DB_USERNAME`、`DB_PASSWORD`、`DB_DATABASE` 配置正确
- [ ] PostgreSQL 服务运行中，可连接
- [ ] Redis 服务运行中，`redis-cli ping` 返回 `PONG`
- [ ] pgvector 扩展已安装（`CREATE EXTENSION IF NOT EXISTS vector;` 执行成功）
- [ ] 所有 `@buildingai/*` 内部包已构建（`dist/` 目录存在）
- [ ] `fs.inotify.max_user_watches` ≥ 65536 或 `CHOKIDAR_USEPOLLING=true` 已设置
- [ ] `node_modules` 已安装（`pnpm install` 已执行）

### 4.3 依赖管理策略

1. **内部包构建顺序**：`@buildingai/config` 和 `@buildingai/types` 被多个包依赖，应优先构建
2. **不要构建 `@buildingai/web-vue`**：它使用 tsup 打包 Vue SFC 组件，需要 Vue 编译器。Nuxt 开发模式直接引用源码
3. **修改内部包源码后**：必须重新运行 `pnpm run build --filter "@buildingai/<包名>"` 才能被 API 识别
4. **pnpm workspace 协议**：所有内部依赖使用 `workspace:*` 协议，确保引用本地版本

### 4.4 代码修改注意事项

本次启动过程中对源代码做了以下修改，这些修改会影响后续开发：

| 文件 | 修改内容 | 影响范围 |
|------|---------|---------|
| `packages/@buildingai/db/src/seeds/seeders/extension.seeder.ts` | zhparser 失败不抛异常 | 开发环境可跳过中文分词 |
| `packages/@buildingai/db/src/seeds/seeders/extension.seeder.ts` | extensions.json 缺失返回空配置 | 无扩展时仍可启动 |
| `apps/web-nuxt/nuxt.config.ts` | 添加 Vite polling 配置 | 开发环境文件监视器优化 |
| `node_modules/...` (Nuxt CLI) | fs.watch 容错处理 | 仅限 ENOSPC 环境 |

**注意：** `node_modules` 中的修改在 `pnpm install` 后会丢失，需要重新应用。建议在无法修改系统限制的环境中保留此 patch。

| 端口 | 服务 | 说明 |

### 5.3 常用命令速查

```bash
# 依赖安装
pnpm install

# 构建所有内部包
pnpm run build --filter "@buildingai/*"

# 启动 API 后端
CHOKIDAR_USEPOLLING=true pnpm run dev:api

# 启动 Web 前端
CHOKIDAR_USEPOLLING=true pnpm run dev:web

# 同时启动 API 和 Web
CHOKIDAR_USEPOLLING=true pnpm run dev:main

# 验证 API 健康状态
curl http://localhost:4090/consoleapi/health

# 检查 PostgreSQL 连接
pg_isready

# 检查 Redis 连接
redis-cli ping

# 查看文件监视器限制
cat /proc/sys/fs/inotify/max_user_watches
```
